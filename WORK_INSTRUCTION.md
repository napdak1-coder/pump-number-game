## 작업 규칙

### 1. 필수 사전 분석
- 구현 요청 시 **요구사항을 구조적으로 먼저 분석**할 것
- 선택 가능한 방식과 각각의 한계를 미리 설명한 후 진행
- 기존 구조에 기능을 억지로 얹지 말고, **요구사항에 맞는 올바른 구조를 처음부터 선택**

### 2. 추측 금지 — 측정 기반 작업
- 추측 기반으로 코드 수정하지 말 것
- **반드시 측정 → 로그 → 수치 확인 후 수정**
- 수치 없이 보고하지 말 것

### 3. 실제 브라우저 검증 필수
- Playwright 등 자동화 테스트가 통과해도 실제 브라우저에서 동작 안 할 수 있음
- **실제 브라우저 기준으로 검증되지 않은 것은 완료로 보고하지 말 것**
- `headless=False` 모드로 실행하여 실제 화면에서 확인

### 4. 반복 수정 3회 룰
- 같은 문제를 **3회 이상 반복 수정**하게 될 경우 즉시 멈출 것
- **근본 원인과 구조적 해결책을 먼저 보고**

### 5. 성능 문제 처리
- 추측으로 고치지 말 것
- **테스트 → 측정 → 수정 → 재측정** 루프로 진행
- 병목은 반드시 **수치로 확인 후 보고**

### 6. AI 모델 사용 시
- rembg, SAM, ONNX 등 사용 시 소재/이미지 유형별 한계를 미리 파악하고 말할 것
- 의류 색상/디테일 손상 여부는 **원본 대비 결과물을 직접 비교**

### 7. UI 기능 구현
- 실제 사용자 동작(클릭, 드래그, 입력 없이 시작 등) 기준으로 **엣지케이스를 미리 검토하고 반영**
- 구현 후 반드시 직접 테스트

### 8. 작업 완료 후 검증
- **서버 재시작 + curl + 브라우저 자동화 교차 검증**을 항상 수행
- 테스트 없이 보고하지 말 것

### 9. 이전 작업 누락 방지
- 구현 요청 시 **이전 대화에서 미완료된 요청사항이 있는지 먼저 확인**
- 누락된 항목은 새 요청과 함께 반드시 포함해서 처리
- 요청이 여러 개 겹칠 때 하나만 처리하고 나머지를 빠뜨리지 말 것

### 10. 코드 블록 필수
- 전달하는 모든 지시문은 **반드시 코드블록으로 복사 가능하게 작성**

## 작업 흐름

```
1. 팀 리더(오퍼스)가 요구사항 분석 및 구조 설계
2. 에이전트(오퍼스)에게 태스크 분배
3. 각 에이전트가 구현 + 단위 테스트
4. 팀 리더가 코드 리뷰 + 통합
5. 실제 브라우저에서 교차 검증
6. 완료 보고 (수치 포함)
```
